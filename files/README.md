# Hyper Render v23 — نسخة محسّنة ومؤمّنة

نفس وظيفة السيرفر الأصلي (v22) بالضبط — نفس الـ endpoints، نفس شكل الطلب والرد،
نفس القوالب البصرية (منشاوي/باسط/دوسري/الافتراضي) — لكن بمعمارية مُعاد هيكلتها
وتحسينات حقيقية في الأداء والأمان والاستقرار.

## هيكل المشروع

```
server.js              نقطة الدخول: express, routes, rate limiting
config.js               كل الإعدادات في مكان واحد (متغيرات بيئة)
lib/
  auth.js               تحقق مفتاح API
  cache.js               كاش مدد الصوت + كاش خلفيات الفيديو المُعاد تحجيمها
  cleanup.js              تنظيف دوري لملفات الفيديو الناتجة القديمة
  fonts.js                تحميل وتسجيل الخطوط عبر fontconfig
  frame.js                بناء الـ SVG الكامل لكل فريم ورندره بـ sharp
  jobs.js                 إدارة الـ jobs + طابور الرندرة (p-queue)
  logger.js               لوجينج منظّم (JSON)
  render.js                المُنسّق الرئيسي لعملية الرندرة الكاملة
  security.js              تحقق الروابط + تحميل آمن + منع path traversal
  svgUtils.js               أدوات SVG المشتركة (فلاتر، overlays، تقسيم نص)
  validation.js              التحقق من صحة طلب /render
  templates/
    defaultTemplate.js       تخطيط النص الافتراضي (آية + ترجمة + زخرفة)
    playerTemplates.js        قوالب المشغل الثلاثة (منشاوي/باسط/دوسري)
```

## أهم التغييرات عن v22

### 1. الخطوط عبر fontconfig بدل base64 (أكبر تحسين أداء)
في v22 كل فريم (ولو عندك آية بـ 15 كلمة كاريوكي، ده معناه 15+ فريم) كان بيتضمن
فيه الفونت كامل مُرمّز base64 جوه `<style>@font-face`. ده كان بيضخّم حجم كل SVG
وبيبطّئ التحليل والرندرة. دلوقتي:
- الخطوط بتتحمّل مرة واحدة وتتسجل عند fontconfig (`fc-cache`).
- الـ SVG بيشاور على اسم الخط بس (`font-family="Amiri"`) من غير أي تضمين.
- لو fontconfig مش متاح في البيئة (تطوير محلي بدون الحزمة)، فيه fallback تلقائي
  لطريقة الـ base64 القديمة عشان الكود يفضل شغال في أي بيئة.

**لازم تتأكد إن حزمة `fontconfig` متثبتة في الكونتينر** (موجودة في الـ Dockerfile).

### 2. طابور رندرة بسقف تزامن (`p-queue`)
كان أي عدد طلبات بيشتغل فورًا مع بعض من غير حد أقصى — ده كان ممكن يقتل
السيرفر (CPU/RAM) بأي عدد طلبات متزامنة، وكان كمان ثغرة DoS بسيطة.
دلوقتي فيه `RENDER_CONCURRENCY` (افتراضي 2) بيحدد كام رندرة تشتغل مع بعض،
والباقي يستنى في الطابور تلقائيًا.

### 3. أمان
- **Rate limiting** على `/render` (`RATE_LIMIT_MAX` كل 15 دقيقة لكل IP).
- **مفتاح API اختياري** (`RENDER_API_KEY`) — لو محدد، أي طلب لازم يبعته في
  هيدر `x-api-key`.
- **مفيش تسريب لـ stack traces** للمستخدم — بيتسجلوا في اللوج بس، والمستخدم
  بياخد رسالة عامة + `errorRef` (رقم الـ job) يقدر يبعتهولك لو محتاج دعم.
- **تحقق من صحة المدخلات** قبل قبول أي طلب: حد أقصى لعدد الآيات، طول النص،
  طول أسماء الحسابات، والتأكد إن كل الروابط من نطاقات مسموح بها.

### 4. أداء
- تحميل الصوتيات (والصور المرتبطة بقوالب المشغل) **بالتوازي** بدل التتابع.
- **كاش حقيقي** لمدد الصوتيات — مبني على رابط الملف نفسه (مش مسار مؤقت خاص
  بكل job زي الأصلي، اللي كان بيخلي الكاش عديم الفايدة عمليًا).
- **كاش لخلفيات الفيديو المُعاد تحجيمها** — لو نفس رابط الخلفية اتطلب قبل كده،
  بنرجّع النسخة المخزّنة بدل ما نعيد `ffmpeg resize` من الصفر (LRU بسيط بحد
  أقصى `BG_CACHE_MAX_FILES`).

### 5. إصلاح باگ حقيقي كان موجود في v22
رابط خط "Noto Naskh Arabic" كان بيترفض بـ 404 لأن كود التحميل كان بيفك ترميز
الرابط (`decodeURIComponent`) فيتحول `%5Bwght%5D` لأقواس حرفية `[wght]` اللي
GitHub بيرفضها. دلوقتي تحميل الخطوط بيستخدم الرابط زي ما هو، وفك الترميز
(المطلوب لروابط `verses.quran.com`) اتفصل في مسار تحميل الملفات العادية بس.

### 6. استقرار
- `/health` endpoint بيورّي حالة الطابور وعدد الـ jobs المتتبّعة.
- Graceful shutdown (`SIGTERM`/`SIGINT`) بيقفل السيرفر بعد ما يخلّص الطلبات
  الحالية بدل ما يقطعها فجأة.
- تنظيف دوري لملفات الفيديو الناتجة (`OUTPUT_TTL_MS`) وملفات الـ jobs القديمة
  من الذاكرة.

## متغيرات البيئة

| المتغير | الافتراضي | الوصف |
|---|---|---|
| `PORT` | `7860` | البورت |
| `RENDER_HOST` | `yousef891238-render-server.hf.space` | الدومين المستخدم في روابط التحميل النهائية |
| `RENDER_API_KEY` | (فاضي) | لو محدد، لازم كل طلب لـ `/render` و`/status` يبعته في `x-api-key` |
| `RENDER_CONCURRENCY` | `2` | عدد الرندرات المتزامنة |
| `RATE_LIMIT_MAX` | `20` | أقصى طلبات لكل IP كل 15 دقيقة |

## التشغيل

```bash
npm install
RENDER_API_KEY=your-secret-here npm start
```

أو بالـ Docker:

```bash
docker build -t hyper-render .
docker run -p 7860:7860 -e RENDER_API_KEY=your-secret-here hyper-render
```

## ملاحظة مهمة لو بتستخدم Hugging Face Spaces

الـ base image الافتراضي على HF Spaces غالبًا مبنيش فيه `fontconfig`. لازم
تتأكد إن الـ Dockerfile المرفق هو اللي بيتستخدم فعليًا (فيه `apt-get install
fontconfig ffmpeg`) عشان تحسين الخطوط يشتغل؛ لو مش متاح، السيرفر هيرجع
تلقائيًا لطريقة الـ base64 القديمة (أبطأ لكن هيفضل شغال).

## اللي محتاج تعمله بنفسك بعد كده (مش جزء من الكود)

- استبدل قيمة `RENDER_API_KEY` بمفتاح حقيقي سري، وابعته من تطبيقك في هيدر
  `x-api-key` مع كل طلب.
- لو الحمل زاد، فكّر في نقل الطابور لـ Redis/BullMQ عشان تقدر تشغّل أكتر من
  نسخة سيرفر مع بعض (الطابور الحالي في الذاكرة، يعني محدود بعملية واحدة).
- فكّر ترفع الفيديوهات النهائية على S3/Cloudinary بدل التخزين المؤقت المحلي،
  خصوصًا لو الـ HF Space بيعمل restart دوري وبيمسح كل حاجة.
